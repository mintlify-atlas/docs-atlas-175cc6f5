GLFM is a C API for mobile app development with OpenGL ES. Inspired by [GLFW](https://github.com/glfw/glfw), it provides a minimal, event-driven interface for display setup, touch and keyboard input, hardware sensors, and application lifecycle across iOS, tvOS, Android, and WebGL (via Emscripten). You implement `glfmMain()` as your entry point and register callbacks — GLFM takes over the platform run loop so you don't have to.

## Platform support

GLFM targets four platforms from a single C codebase. Metal is additionally available on Apple platforms.

## Key features

  <Card title="OpenGL ES 2, 3, and Metal" icon="display">

## Non-goals

GLFM is intentionally minimal. It will never provide:

- Image loading
- Text rendering
- Audio playback
- Menus, UI toolkits, or scene graphs
- Integration with platform features like web views, maps, or game center

Use GLFM alongside other cross-platform C libraries for those capabilities.

## Frequently asked questions

<AccordionGroup>
  <Accordion title="Why is the entry point glfmMain() and not main()?">
    On iOS, the `main()` function must create an autorelease pool and call `UIApplicationMain()`, which never returns. GLFM's own `main()` handles that setup and only calls your `glfmMain()` after the `UIApplicationDelegate` and `UIViewController` are fully initialized. Using a standard `main()` would make iOS support impossible.
  </Accordion>
  <Accordion title="Why is GLFM event-driven? Why does GLFM take over the main loop?">
    The same reason as above: `UIApplicationMain()` on iOS and the browser's event loop on WebGL are both non-blocking run loops that GLFM must integrate with. Taking control of the main loop is the only portable way to support all target platforms.
  </Accordion>
  <Accordion title="What IDE should I use?">
    Use **Xcode** for iOS and tvOS targets, and **Android Studio** for Android. For everyday development without a mobile simulator, consider using [GLFW](https://github.com/glfw/glfw) on the desktop and porting to GLFM later. For maximum OpenGL portability, target OpenGL 3.2 Core Profile on desktop and OpenGL ES 2.0 on mobile.
  </Accordion>
  <Accordion title="Is GLFM thread-safe?">
    No. All GLFM functions must be called on the main thread — from `glfmMain()` or from GLFM callback functions.
  </Accordion>
</AccordionGroup>
