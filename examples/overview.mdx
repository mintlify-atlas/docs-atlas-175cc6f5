The `examples/` directory in the GLFM repository contains self-contained C programs that each demonstrate a distinct area of the API. Reading them alongside the header is the fastest way to understand how GLFM's event-driven model works in practice — from the `glfmMain` entry point through surface lifecycle management, input handling, and frame rendering.

## Building the examples

The examples share a single `CMakeLists.txt` and are gated behind the `GLFM_BUILD_EXAMPLES` flag. See the platform-specific build guides for full instructions:

- **Xcode (iOS / tvOS / macOS)** — run `cmake -D GLFM_BUILD_EXAMPLES=ON -B build/apple -G Xcode`, then open `build/apple/GLFM.xcodeproj` and select the target you want (e.g. `glfm_touch`).
- **Emscripten (Web)** — run `emcmake cmake -D GLFM_BUILD_EXAMPLES=ON -B build/emscripten && cmake --build build/emscripten`, then serve with `emrun build/emscripten/examples`.
- **Android Studio** — add `externalNativeBuild` pointing at the root `CMakeLists.txt` with `-DGLFM_BUILD_EXAMPLES=ON`, and reference the assets directory (see below).

Each example compiles to a library named `glfm_<example>` (e.g. `glfm_triangle`, `glfm_touch`).

## The assets directory

Some examples (`typing`, `compass`, `shader_toy`, `test_pattern`) load files at runtime from `examples/assets/`. On Android, the Gradle `sourceSets.main.assets.srcDirs` must point to this directory:

```gradle
sourceSets.main {
    assets.srcDirs = ["../../../examples/assets"]
}
```

On Emscripten the build system packages the assets automatically. On iOS and macOS they are copied into the app bundle by CMake.

<Note>
  Examples that use `file_compat.h` (a small portability helper bundled in `examples/deps/`) require `FILE_COMPAT_ANDROID_ACTIVITY` to be defined before the include so the helper can reach the Android asset manager through GLFM's `glfmGetAndroidActivity`.
</Note>
